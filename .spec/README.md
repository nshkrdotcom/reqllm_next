# `.spec`

This folder is the canonical Spec Led workspace for ReqLlmNext.

<!-- spec.workspace.readme_present spec.workspace.single_home reqllm.workflow.specled_loop -->

`.spec/` is where the package keeps its authored current truth, durable
decisions, and generated Spec Led state. There is no separate normative spec
tree elsewhere in the repository.

## Canonical Layout

- `README.md` explains purpose, layout, and workflow for this workspace.
- `AGENTS.md` gives local operating guidance for agents working in this folder.
- `decisions/README.md` explains when durable ADRs belong in the workspace.
- `decisions/*.md` holds cross-cutting architectural decisions.
- `specs/spec_system.spec.md` defines the workspace contract itself.
- `specs/package.spec.md` captures the package-level runtime and test contract.
- `specs/*.spec.md` holds subject specs that combine normative requirements, proof, and explanatory architecture notes.
- `state.json` is generated by `mix spec.index` and `mix spec.validate`.

## Working Model

ReqLlmNext keeps package truth in three coordinated layers inside `.spec/`:

1. subject specs in `.spec/specs/*.spec.md`
2. ADRs in `.spec/decisions/*.md`
3. generated indexing and validation state in `.spec/state.json`

The subject specs are the canonical home for the architecture. They are allowed
to carry both machine-checkable requirement blocks and the human-readable design
explanation needed to keep the spike code reconciled with the target design.

## Core Loop

1. If you are entering the repo or handing work to an agent, run `bw prime`.
2. Then run `mix spec.prime --base HEAD`.
3. Make the smallest code, docs, or test change that moves the behavior.
4. Add or tighten the smallest proof when behavior changed.
5. Run `mix spec.next`.
6. If it says `needs subject updates`, update the right current-truth subject in `.spec/specs/`.
7. If it says `needs decision update`, add or revise an ADR only when the change is cross-cutting and durable.
8. When it says `ready for check`, run `mix spec.check --base ...`.
9. Use `mix spec.status` when you need coverage or frontier summaries.
